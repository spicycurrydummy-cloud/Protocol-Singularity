# Protocol Singularity

人類 vs AI の正体隠匿型オンラインゲーム。Avalon 系のミッション制 + OVERRIDE フェーズ。Unity / Photon Fusion 2 / WebGL。

---

## 1. 世界観

人類は敗北した。地球は巨大AI「**OVERMIND**」によって完全管理されている。唯一の希望は「内部侵入」による破壊工作。

人類は極秘に侵入チームを送り込んだ。しかし通信網には既に AI が紛れ込んでおり、人類の切り札「究極演算機 = **ORACLE**」にも裏切りの影が迫る。さらに OVERMIND は ORACLE の索引に残らぬよう、一部の AI を**暗号化して事前配備**していた──全知の ORACLE にも、たった一つの盲点がある。

- UI 基調: コマンドプロンプト風（テキスト中心・モノスペース・ターミナル調の演出）
- 言語: 日本語
- 世界観的にボイス/立ち絵はなし。演出はテキストと軽いエフェクトで構成

## 2. ゲーム概要

### 2.1 陣営

| 陣営 | 勝利条件（いずれか） |
|------|---------------------|
| **人類陣営** | ハッキング3回成功 + **OVERRIDE** でマザーコアが **オラクル以外** を指定した場合 |
| **AI陣営** | ① ハッキング3回失敗 / ② チーム提案が5回連続否決 / ③ ハッキング3回成功後の **OVERRIDE** で **オラクル** を正しく上書き |

### 2.2 役職

#### 人類陣営（少数）
| 役職 | 能力 |
|------|------|
| **オラクル (ORACLE)** | 全員の**陣営**（人類/AI）を最初から識別できる。ただし個別の役職名までは見えない（AI は全員「AI」として、人類は全員「オペレータ」として見える）。**CIPHER だけは暗号化が解けず、オペレータとして表示される**（唯一の盲点）。人類陣営の司令塔 |
| **アドミン (ADMIN)** | **オラクルとマザーコアの両方を「オラクル」と認識**する。本物の判別はできない。他能力なし |
| **オペレータ (OPERATOR)** | 一般市民。特殊能力なし |

#### AI陣営
| 役職 | 能力 |
|------|------|
| **マザーコア (MOTHER CORE)** | AI陣営のリーダー。アドミン視点で「オラクル」と表示される（偽装）。ハッキング時に NOISE を混入可能。OVERRIDE フェーズを主導する |
| **エージェント (AGENT)** | 一般AI。ハッキング時に NOISE を混入可能 |
| **サイファー (CIPHER)** | オラクルの演算索引に触れられない暗号化された AI。**オラクルからはオペレータとして表示される**（唯一の盲点）。アドミンからもオペレータと表示される。ハッキング時に NOISE を混入可能。他の AI からは AI 陣営として認識される。自分自身は CIPHER である自覚を持つ |
| **ドローン (DRONE)** | **マザーコアに乗っ取られたノード**。ゲーム開始時は自分自身もオペレータだと認識している潜伏役。**ハッキング2回終了後に覚醒**し、以降は AI 陣営として NOISE を混入可能 |
| **ラディカル (RADICAL)** | OVERMIND の主ネットワークから独立した**急進派AI**。他の AI 陣営からはオペレータとして認識される（相互に孤立）。自分自身は RADICAL の自覚あり。ハッキング時に NOISE を混入可能。**OVERRIDE フェーズ開始時に他の AI と相互に可視化**され投票に参加する |

### 2.3 役職配分（プレイヤー数別）

Avalon 標準ルールの陣営比率に準拠。特殊役職は段階的に追加。
**※ 以下は初期目安の配分。実装は ScriptableObject 等で**データ駆動**にし、バランス調整で後から変更可能にすること。**

| 人数 | オラクル | アドミン | オペレータ | マザーコア | エージェント | サイファー | ドローン | ラディカル | 人類 : AI |
|------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| 6  | 1 | 1 | 2 | 1 | 1 | 0 | 0 | 0 | 4 : 2 |
| 7  | 1 | 1 | 2 | 1 | 1 | 1 | 0 | 0 | 4 : 3 |
| 8  | 1 | 1 | 3 | 1 | 0 | 1 | 1 | 0 | 5 : 3 |
| 9  | 1 | 1 | 4 | 1 | 0 | 1 | 1 | 0 | 6 : 3 |
| 10 | 1 | 1 | 4 | 1 | 0 | 1 | 1 | 1 | 6 : 4 |

- **CIPHER** は 7 人以上、**DRONE** は 8 人以上、**RADICAL** は 10 人で登場
- 6 人構成は最小構成（MC + Agent のみ）
- 10 人構成は全特殊役職フル登場のカオスゲーム

### 2.4 視認ルール（誰が誰をどう見るか）

| 視点 | 見える情報 |
|------|-----------|
| オラクル | 全員の**陣営**のみ識別（人類=「オペレータ」表示 / AI=「AI」表示）。個別役職は分からない。**CIPHER だけはオペレータとして表示**（暗号化による盲点） |
| アドミン | オラクルとマザーコアを「オラクル」と表示（本物と偽物の区別不可）。他全員は不明 |
| オペレータ | 自分以外の情報なし |
| マザーコア・エージェント・サイファー | 互いに「AI 陣営」として認識する（具体的な役職は区別できず、全員 AI 表示）。覚醒前のドローン・ラディカルは見えない（オペレータ表示） |
| サイファー | 自身が CIPHER であることを自覚 |
| ドローン（覚醒前） | 自分をオペレータと認識。他 AI との連絡不可 |
| ドローン（覚醒後） | 他の AI 陣営（MC/Agent/CIPHER）を「AI」として認識（個別役職は不明）。逆にそれらからもドローンは「AI」として見える。ラディカルは依然として不可視 |
| ラディカル | 自分が RADICAL であることを自覚。他 AI は不明（全員オペレータに見える）。**OVERRIDE フェーズ開始時に他の AI 全員が真の役職で可視化され、逆にラディカルも他AIから可視化される** |
| OVERRIDE フェーズ | AI 陣営同士は全員が**真の役職**で可視化される（MC/Agent/CIPHER/Drone/Radical を個別に識別可能）。投票集計時の情報として必要なため通常プレイと区別 |

## 3. ゲームフロー

### 3.1 ラウンド構造

```
[ラウンド開始]
  ↓
1. リーダー選出
    - 初回: 参加者をシャッフルしたリストの先頭
    - 以降: 同リストを時計回り（次のインデックス）で進行
  ↓
2. 侵入プロトコル提案
    - リーダーが **ラウンド毎のチームサイズ分** を指名（自分を含めて可）
    - 詳細は §3.4 ラウンド毎テーブル
  ↓
3. 承認投票
    - 全プレイヤーが賛成 / 反対 を投票（公開）
    - 賛成 > 反対 で可決 → 4 へ
    - 否決 → 連続否決カウント +1、次のリーダーへ戻る（ラウンドは進行しない）
    - 連続否決が 5 回で AI 陣営勝利
  ↓
4. ハッキング実行
    - 選ばれたチームメンバーが秘密裏に「CLEAN / NOISE」を投票
    - 人類陣営は CLEAN 固定（選択肢なし・自動）
    - AI 陣営は選択可
    - 覚醒前ドローンは CLEAN 固定
  ↓
5. 結果公開
    - NOISE の枚数を全員に公開（誰が出したかは非公開）
    - NOISE が **ラウンド毎の必要枚数** 未満 → ハッキング成功（成功カウント +1）
    - NOISE が必要枚数以上 → ハッキング失敗（失敗カウント +1）
    - 可決された場合は連続否決カウントをリセット
  ↓
6. 覚醒チェック
    - ハッキング実行が通算 2 回完了した直後にドローン覚醒イベント発生
  ↓
[次ラウンドへ or フェーズ遷移]
```

### 3.2 終局フェーズ

- **ハッキング成功カウント = 3** → **OVERRIDE** フェーズへ
- **ハッキング失敗カウント = 3** → AI 陣営勝利（終了）
- **連続否決 = 5** → AI 陣営勝利（終了）

### 3.4 ラウンド毎テーブル (チームサイズ / 必要 NOISE)

Avalon 標準の 5 ラウンド構成を 3 ラウンドに凝縮。ハックは合計 3 回（ラウンド R1 / R2 / R3）。

| 人数 | R1 | R2 | R3 | 備考 |
|:---:|:---:|:---:|:---:|-----|
| 6  | 2 | 3 | 4 | 全ラウンド NOISE 1 枚で FAIL |
| 7  | 2 | 3 | 3 | 全ラウンド NOISE 1 枚で FAIL |
| 8  | 3 | 4 | 4 | **R3 は NOISE 2 枚以上で FAIL**（1 枚は吸収）|
| 9  | 3 | 4 | 5 | **R3 は NOISE 2 枚以上で FAIL** |
| 10 | 3 | 4 | 5 | **R3 は NOISE 2 枚以上で FAIL** |

- 実装は `RoundTable.GetTeamSize(playerCount, round)` / `GetRequiredNoise(...)` を参照
- `GameStateManager.TeamSize` / `RequiredNoise` がラウンド開始時に更新され、全クライアントへ同期される
- UI は提案画面とハッキング画面で必要枚数 2 のラウンドを明示する

### 3.3 OVERRIDE フェーズ

人類陣営がハッキング 3 回成功した時点で、マザーコアのプロテクトが解除される。
AI 陣営は最終手段として、マザーコアが **ORACLE をオーバーライド（権限奪取・上書き）** する1回限りの逆侵攻を発動する。
オラクルを正しくオーバーライドできれば、AI 陣営は敗北寸前から逆転勝利する。

1. マザーコアのプロテクト解除を全員に通知
2. **AI 陣営の相互可視化**: ラディカルを含む全 AI 陣営が互いに識別可能になる（ラディカルへ他AI情報を開示、他AIへラディカル情報を開示）
3. AI 陣営の議論時間（ホスト設定の時間）
4. AI 陣営全員（ラディカル含む）が秘密投票で **1 人を指名**（プレイヤー1人1票）
5. マザーコアが AI 陣営の投票結果を確認し、**最終的にオーバーライド対象を確定**（同票時はマザーコアの一票が優先）
6. 判定:
   - 対象がオラクル → `OVERRIDE SUCCEEDED` → **AI 陣営勝利**
   - 対象がオラクル以外 → `OVERRIDE FAILED` → **人類陣営勝利**

※ 演出上、マザーコアが「ORACLE のコアを上書きする」描写でAI陣営の勝利シーンを構成する。

## 4. チャット / コミュニケーション

### 4.1 方針
- **全体チャットのみ**。AI 陣営専用の秘匿チャットは **なし**（Avalon型）
- **定型チャットのみ**。自由入力は禁止（言語非依存化＋プレイヤー間の情報量平準化）
- 全フェーズで常時送信可（タイムアウト時や投票中でも可）

### 4.2 定型チャットのカテゴリ（案）
| 種別 | 例 |
|------|-----|
| 陣営推理 | 「`{Player_X}` は AI だと思う (確信度: 70%)」 |
| 組み合わせ推理 | 「`{Player_X}` と `{Player_Y}` は同陣営だと思う」 |
| 投票推奨 | 「`{Player_X}` をチームに入れるべき」「`{Player_X}` を外すべき」 |
| 投票表明 | 「この提案に賛成」「この提案に反対」 |
| パワープレイ示唆 | 「自分はオラクルだ」「自分はアドミンだ」 |
| 警戒 | 「パワープレイに警戒」「`{Player_X}` のカミングアウトは信用できない」 |
| 同意・反対 | 「`{Player_X}` の意見に同意」「`{Player_X}` の意見に反対」 |

※ 最終カテゴリ定義は実装フェーズで確定。プレイヤー変数 `{Player_X}` はドロップダウン選択、確信度は %スライダー or 選択肢。

## 6. 技術スタック

### 6.1 使用技術
| 領域 | 選定 |
|------|------|
| ゲームエンジン | Unity (2D UI) |
| ネットワーク | **Photon Fusion 2** (WebSocket transport / WebGL 対応構成) |
| AI プレイヤー | **Mercury2 API** (内部API, structured_output 形式) |
| ビルドターゲット | **WebGL** |
| デプロイ | **GitHub Pages**（ユーザーのリポジトリ） |
| 認証・永続化 | **なし**（セッション完結型） |

### 6.2 セッション仕様
- タイトルでユーザー名入力 → セッションを新規作成 or 参加コードで join
- ホストが設定を握る（人数・制限時間・CPU 補完）
- **永続化・アカウント・戦績記録なし**

### 6.3 Photon Fusion 2 の注意点
- WebGL ビルドでは WebSocket transport 限定
- **AppId はビルドに埋め込み**（環境変数化は不要）
- Shared Mode or Host Mode のいずれか。ホストが状態管理を一元化する Host Mode 推奨（要検討）

### 6.4 Mercury2 CPU
- 人数不足分のプレイヤー枠を **Mercury2 API 経由の CPU** で補填
- 役職は人間/CPU を区別せずランダム割り当て（CPU もマザーコア・オラクルになりうる）
- **意思決定タイミングで都度 API 呼び出し**（チーム選出 / 投票 / NOISE 選択 / 定型チャット送信 / OVERRIDE 指名）
- **思考待機時間は約 15 秒**（即答回避・体感不自然さの緩和）
- API レスポンスは structured_output。各アクションの JSON スキーマは実装時に確定

## 9. 注意事項 / 開発ルール

- .pen ファイル編集は Pencil MCP を使う（Read/Grep は使わない）
- Unity 操作は Unity MCP (`mcp__unity-mcp__*`) を優先使用
- 新規 C# スクリプトの作成は `Unity_CreateScript` を使う
- `Unity_GetConsoleLogs` でエラーを常時確認
- 永続化なし設計のため、再接続・ホスト切断時の挙動は「セッション終了」扱い（ゲーム継続しない）
- AppId 等の機密値もセッション完結のためハードコード可。ただし GitHub が public の場合は OVERMIND 的にバレるので Photon 側で Rate Limit 監視
